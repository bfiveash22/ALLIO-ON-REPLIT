I've completed the investigation into the "stuck assignments" scenario and I found a critical deadlock bug in the scheduler that was bottlenecking the entire Agent Network.

**What was happening:**
The scheduler limits concurrent executions to 8 tasks (`MAX_CONCURRENT_TASKS`). Sometimes, tasks handed off to external API endpoints (like OpenAI) would hang indefinitely due to network drops without ever throwing an error. This left them perpetually marked as "executing" in the agent's memory. When 8 tasks got hung up like this, the entire system locked up! 
Worse, because the active execution queue was full, the scheduler stopped processing new queue checks—which completely bypassed the safety mechanism that was supposed to reset stuck tasks in the database.

**The Fix:**
1. **Safety Clearance Restored:** I moved the database clearance logic to run *before* the max-capacity check. This ensures that even when the system believes its active queues are full, it still checks the database to reset old tasks that have been hanging for 2+ hours.
2. **Infinite Loop Protection:** I wrapped the `executeAgentTask()` function in a robust 30-minute `Promise.race` timeout. Now, if any agent task hangs for over 30 minutes, the orchestrator forcibly terminates the execution, fails it gracefully, and frees up the slot.

This completely eliminates the deadlock scenario that was stopping new tasks from processing. The pipeline should reliably handle heavy loads without freezing now!
