## 1. Environment and Targeting
- **CRITICAL DEPLOYMENT TARGET:** This codebase runs on a Linux VPS via PM2, but development often occurs on Windows.
- **Rule (File Execution):** Only use `deploy.ps1` for pushing code. DO NOT run arbitrary deploy scripts like `vps-deploy.ts` or `deploy-remote.sh`. All old variants have been explicitly deleted.
- **Rule (Nginx & PM2 Safety):** `deploy.ps1` contains safe restart mechanisms. NEVER rewrite `deploy.ps1` to inject Nginx here-doc overrides (e.g. `cat > /etc/nginx/sites-available/...`) as this wipes out the production SSL/Domain configs.
- **Rule (Text Encoding):** If PowerShell bulk string replacements are utilized, they silently default to `UTF-16LE` encoding, crashing the Linux VPS (`tsc` and PM2 execution). You MUST specify `UTF-8` or use Node (`tsx`) script replacements.
- **Paths and Zips:** Do not bake `\` backslash paths into output binaries. When writing archive/zip scripts that will be unzipped on Linux, make sure to normalize ALL paths to use `/` forward slashes before injecting them into the archive! Otherwise, `unzip` on Linux creates literal backslash files.

## 3. AGENT REGISTRY & DATABASE INTEGRITY (CRITICAL)
- **SINGLE SOURCE OF TRUTH:** `shared/agents.ts` is the ONLY definitive list of authorized agents (exactly 44 agents as of Phase 5).
- **ANTI-HALLUCINATION RULE:** NEVER hardcode agent names in other services (e.g. `sentinel.ts`, `sentinel-orchestrator.ts`). All backend orchestration MUST dynamically calculate divisions and agent counts directly from `shared/agents.ts`.

- **SEEDBING:** If you see "Ghost Agents" in the database (`agent_registry`), you MUST wipe the PostgreSQL table to force Sentinel Orchestrator to natively re-seed.

## 4. Webhooks & OpenClaw
